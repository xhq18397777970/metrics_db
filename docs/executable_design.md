# 集群指标库可执行设计文档

## 1. 目标

构建一个持续运行的集群指标采集系统，通过现有 `tools` 中的 API 接口，自动采集所有集群在全时间段内的指标数据，并按 `5 分钟` 粒度写入时序数据库，后续支持：

- 查询任意集群在任意时间段内的指标
- 查询 CPU/QPS/HTTPCode/TP 的历史基线
- 支持长期保留数据
- 支持未来增加新的指标工具而不推翻当前架构

本设计文档强调两点：

- 每个阶段都必须可单独验证
- 当前阶段验证通过后，才进入下一阶段

## 2. 当前现状

当前仓库内已有如下基础材料：

- 集群清单：`cluster.json`
- 指标工具：`tools/cpu.py`、`tools/qps.py`、`tools/code.py`、`tools/tp.py`

已知现状：

1. `cluster.json` 当前共 `70` 个集群，而不是 80 个
2. 工具返回值风格不统一
3. 部分工具鉴权逻辑重复
4. `cpu.py` 当前入口函数仍是硬编码示例，不适合直接纳入调度链路

这意味着第一阶段不能直接上调度和落库，必须先做“采集器标准化”。

## 3. 总体设计

### 3.1 核心原则

1. **采集标准化**：所有工具必须被包装成统一 Collector 接口
2. **存储标准化**：数据库中采用统一的指标点模型，不为每新增一个工具改一次表
3. **执行可重入**：同一时间窗重复执行不能产生脏数据
4. **部分失败可恢复**：单个工具失败不能阻塞整个集群的其他指标入库
5. **先原始点，后聚合视图**：原始指标点和基线聚合分层处理

### 3.2 逻辑架构

```text
cluster.json / clusters表
        |
        v
  Scheduler(每5分钟生成一个采集时间窗)
        |
        v
 Dispatcher(按集群拆任务，控制并发、重试、超时)
        |
        v
 Collector Registry(注册所有采集器)
        |
        +--> CPU Collector
        +--> QPS Collector
        +--> HTTPCode Collector
        +--> TP Collector
        +--> Future Collectors...
        |
        v
 MetricPoint 标准化结果
        |
        v
 TimescaleDB
   - metric_points
   - collection_runs
   - collection_failures
   - continuous aggregates
        |
        v
 Query Service / Baseline Service
```

### 3.3 模块划分

建议仓库后续按如下结构演进：

```text
docs/
  executable_design.md
src/
  config/
  domain/
  collectors/
  orchestrator/
  storage/
  services/
  api/
tests/
sql/
```

说明：

- `collectors/`：每个指标工具对应一个 collector 适配器
- `orchestrator/`：调度、并发控制、重试
- `storage/`：TimescaleDB 仓储和 SQL
- `services/`：基线计算和查询服务
- `api/`：后续对外提供查询接口
- `tests/`：每个阶段对应的测试集

## 4. 数据模型

### 4.1 为什么不用宽表

当前指标看起来只有 CPU、BPS、HTTPCode、TP，但未来还会增加新工具。  
如果一开始设计成宽表：

```text
cluster_name | ts | cpu | qps | http_2xx | http_4xx | http_5xx | tp | ...
```

后续每加一个工具，都要改表结构、改入库逻辑、改查询逻辑，演进成本高。

因此建议原始层采用“窄表”：

### 4.2 原始指标点表 `metric_points`

建议字段：

- `bucket_time`：5 分钟桶起始时间
- `window_start`
- `window_end`
- `cluster_name`
- `metric_name`
- `metric_value`
- `labels`：JSONB，可存方向、HTTPCode 分类等附加标签
- `source_tool`
- `collect_status`
- `collected_at`

说明：

- `cpu_avg` 存为一条指标
- `net_bps` 存两条指标，`labels.direction=in/out`
- `http_code_count` 存三条指标，`labels.class=2xx/4xx/5xx`
- `tp_avg` 存为一条指标
- 后续新增工具只需增加新的 `metric_name`

### 4.3 任务执行表 `collection_runs`

建议字段：

- `run_id`
- `bucket_time`
- `cluster_name`
- `collector_name`
- `status`
- `retry_count`
- `started_at`
- `finished_at`
- `error_message`

用途：

- 判断某个时间窗是否采集成功
- 支持重试和补数
- 为后续排障提供证据

### 4.4 基线查询层

原始表不直接承担全部基线查询压力。  
应基于原始表建立连续聚合或汇总表，例如：

- `metric_points_5m`
- `metric_points_1h`
- `metric_points_1d`

基线查询优先使用 `5m` 或 `1h` 聚合层，而不是全扫原始表。

## 5. 关键技术决策

### 5.1 数据库

选型：`TimescaleDB`

理由：

- 原生适合时间序列写入
- 适合长期保留
- 支持连续聚合，便于后续做基线
- 保留 SQL 查询能力，便于做“上周同时间段基线”“历史 10:00-13:00 CPU 基线”

### 5.2 并发模型

第一版建议使用：

- `asyncio` 统一调度
- 同步工具先通过线程池包装
- 全局并发限制
- 对单一外部接口做限流

原因：

- 当前规模只有 70 个集群
- 先追求稳定性和可测性，不先引入 MQ
- 后续如果扩容，再演进为“调度器 + Worker + 队列”

### 5.3 配置源

第一版继续使用 `cluster.json` 作为集群配置源。  
后续如果集群清单变化频繁，再迁移到数据库配置表。

## 6. 分阶段实施方案

---

## 阶段 0：冻结指标契约

### 目标

明确每个工具输入、输出、错误语义，形成统一契约。

### 产出

- 每个工具的入参定义
- 每个工具的标准输出定义
- `metric_name` 命名规范
- 时间窗统一格式

### 需要处理的问题

- `cpu.py` 需要改为可传入集群和时间窗
- `qps.py` 当前使用毫秒时间戳
- `code.py` 和 `tp.py` 当前使用字符串时间
- 所有工具都需要统一为同一种时间窗输入模型

### 测试方法

对每个工具执行一次“单集群、单时间窗”的验证。

验证项：

- 正常返回时能得到标准化结果
- 无数据时能得到明确错误
- 接口异常时能得到明确错误

### 通过标准

- 4 个工具都能映射为统一的数据结构
- 无硬编码集群名和时间
- 所有错误都能被上层识别

### 未通过时不得进入下一阶段

- 任一工具仍依赖示例参数
- 返回结构不能稳定解析

---

## 阶段 1：实现 Collector 标准层

### 目标

把每个工具包装成统一的 Collector 接口。

### 产出

- `BaseCollector` 或统一协议
- `CpuCollector`
- `QpsCollector`
- `HttpCodeCollector`
- `TpCollector`
- `CollectorResult`
- `MetricPoint`

### 设计要求

每个 Collector 必须满足：

- 输入：`cluster_name + window_start + window_end`
- 输出：`list[MetricPoint]`
- 工具异常时返回可记录的失败结果，而不是直接让进程崩溃

### 测试方法

使用 mock 替代真实 API 调用。

测试场景：

- 成功返回单指标
- 成功返回多指标
- 返回空数据
- 返回错误

### 通过标准

- 每个 collector 都能独立测试通过
- 新增一个 collector 时，不需要修改调度器核心逻辑

### 下一阶段准入条件

- 所有 collector 单元测试通过
- 指标命名无冲突

---

## 阶段 2：实现时间窗与集群加载

### 目标

实现“5 分钟桶”时间窗和集群清单加载逻辑。

### 产出

- 时间窗对齐函数
- `cluster.json` 读取器
- 集群扁平化列表输出

### 关键规则

- 任意采集时刻都必须对齐到固定 5 分钟桶
- 一个时间窗只允许有一个标准表示
- 不能因本地时间误差产生重复桶

### 测试方法

测试时间点：

- `10:00:01` 应归到 `09:55:00-10:00:00` 或 `10:00:00-10:05:00`，必须提前固定规则
- `10:04:59`
- `10:05:00`

集群加载测试：

- 验证当前集群总数为 `70`
- 验证分组信息可追溯

### 通过标准

- 时间窗计算具备确定性
- 集群总数、分组、名称可稳定输出

### 下一阶段准入条件

- 时间窗逻辑测试全部通过
- 集群加载逻辑测试全部通过

---

## 阶段 3：实现调度与并发采集

### 目标

实现一个能对“所有集群 x 所有 collector”发起采集任务的编排器。

### 产出

- `Scheduler`
- `Dispatcher`
- 并发限制器
- 重试策略
- 超时控制

### 执行策略

一个 5 分钟时间窗内：

1. 读取全部集群
2. 为每个集群生成采集任务
3. 对每个集群调用全部 collector
4. 使用受控并发执行
5. 收集成功与失败结果

### 测试方法

使用假的 collector：

- 慢返回 collector
- 失败 collector
- 部分成功 collector

验证项：

- 并发度被限制在配置内
- 某 collector 失败不影响其他 collector
- 某集群失败不影响其他集群
- 同一个任务不会被执行两次

### 通过标准

- 能完整跑完一次 `70 集群 x 全 collector` 模拟任务
- 无崩溃
- 失败可统计
- 输出结果可落库

### 下一阶段准入条件

- 并发编排测试通过
- 日志中可定位失败任务

---

## 阶段 4：实现 TimescaleDB 入库层

### 目标

把标准化指标点稳定写入数据库，并保证幂等。

### 产出

- 建表 SQL
- Hypertable 初始化
- 批量写入接口
- Upsert 逻辑

### 入库要求

- 同一个 `cluster + metric + bucket + labels` 重复写入不产生重复数据
- 支持批量写入
- 支持记录采集状态

### 测试方法

本地数据库验证：

- 插入一批指标点
- 对同一批指标点重复写入一次
- 查询结果确认没有重复
- 插入部分失败数据，确认失败记录可追踪

### 通过标准

- 幂等写入成功
- 重复执行同一时间窗无重复行
- 数据字段能覆盖当前所有工具输出

### 下一阶段准入条件

- 数据库 schema 稳定
- 插入/重复插入测试通过

---

## 阶段 5：完成单时间窗端到端采集

### 目标

完成一次真实的“读取集群 -> 调用工具 -> 标准化 -> 入库”的全链路。

### 产出

- 手工触发入口
- 指定时间窗的端到端执行能力

### 测试方法

选择：

- 1 个集群 + 全部 collector
- 5 个集群 + 全部 collector
- 70 个集群 + 全部 collector

验证项：

- 数据成功落库
- 成功数和失败数可统计
- 某个集群可查询到完整指标

### 通过标准

- 至少一轮真实采集成功落库
- 落库数据条数符合预期
- 日志中能看到完整执行链路

### 下一阶段准入条件

- 端到端链路验证成功
- 失败数据可重试

---

## 阶段 6：接入定时运行与补数

### 目标

实现“每 5 分钟自动采集”与“历史时间窗补数”。

### 产出

- 定时调度入口
- 补数入口
- 重跑指定时间窗能力

### 测试方法

定时验证：

- 连续运行 1 小时
- 检查是否每个 5 分钟桶都产生了任务记录

补数验证：

- 指定过去 12 个时间窗执行补数
- 验证无重复数据

### 通过标准

- 连续运行期间无漏桶
- 补数执行无重复入库
- 失败时间窗能单独重跑

### 下一阶段准入条件

- 调度稳定
- 补数稳定

---

## 阶段 7：实现基线查询能力

### 目标

支持用户查询：

- 某个时间段内的 CPU 基线
- 上周同时间段的 CPU 基线
- 后续扩展到 QPS、HTTPCode、TP

### 产出

- 聚合视图或连续聚合
- 基线查询服务
- 查询接口

### 基线定义建议

必须先固定“基线”的业务定义，例如：

- `avg`
- `p50`
- `p95`
- 过去 7 天同时间窗
- 过去 4 周同星期同时间窗

### 测试方法

测试样例：

- 查询 `10:00-13:00` CPU 基线
- 查询“上周同时间段 CPU 基线”
- 手工 SQL 与服务结果比对

### 通过标准

- 服务结果与手工 SQL 一致
- 查询延迟满足预期
- 查询逻辑不直接全扫原始数据

## 7. 测试策略

### 7.1 测试分层

- 单元测试：collector、时间窗、标准化逻辑
- 集成测试：编排器 + 仓储
- 端到端测试：真实接口 + 数据库

### 7.2 每阶段必须满足

- 有明确输入
- 有明确输出
- 有成功标准
- 有失败判断

若缺一项，则该阶段不允许进入实施状态。

## 8. 风险与控制

### 风险 1：接口不稳定

控制方式：

- collector 层统一错误模型
- 重试和超时
- 失败任务持久化

### 风险 2：并发过大导致上游接口限流

控制方式：

- 全局并发限制
- 按 host 限流
- 后续支持退避重试

### 风险 3：未来新增工具导致代码发散

控制方式：

- 所有新工具只能通过 collector 插件机制接入
- 禁止在调度器中写特例逻辑

### 风险 4：重复执行导致脏数据

控制方式：

- 幂等主键
- upsert
- 每个时间窗执行记录可追踪

## 9. 第一阶段建议从哪里开始

下一步只做一件事：

**先完成阶段 0 和阶段 1。**

原因：

- 目前最大的风险不是数据库，也不是调度
- 而是工具接口契约不统一
- 如果契约不统一，后续并发、落库、基线都会反复返工

建议实施顺序：

1. 统一工具接口输入输出
2. 用 mock 做 collector 单测
3. 确认标准化指标模型
4. 再做调度与入库
